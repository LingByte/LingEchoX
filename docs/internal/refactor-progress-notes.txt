# LingEchoX 复刻 VoiceServer 底层 — 进度跟踪

## 背景
LingEchoX 与 VoiceServer 共用同一 module path: `github.com/LinByte/VoiceServer`。
VoiceServer 底层 SIP 设计更解耦（业务通过 InviteHandler 接口注入，
录音抽到 pkg/voice/recorder 统一 stereo WAV）。
本计划逐层把 VS 底层迁回 LEX，并保留 LEX 独有的业务功能
（conversation / voicedialog / webseat / persist / outbound 业务流程）。

## 总体策略
- 保持 LEX 业务包不动（conversation / voicedialog / webseat / persist），先把底层撑起来。
- 底层（sip/* + voice/*）逐包对齐到 VS，再把业务侧的 import 切回去。
- 每完成一档就跑 `go build ./...`；不允许任何持续编译失败。

- [x] **录音分片上传 + 流式合并 Flush（C 方案）**
        2026-05-16。彻底重构 pkg/voice/recorder 的录音存储路径：
        ① **Session-wide 共享 base + per-leg 持久化 pen**：新增 `legPlacer`
          流式放置器，跨多次 chunker tick 维护连续 per-leg 时间轴；
          `sessionBaseNs` 在第一帧 lazy-set，所有后续 chunk 共享 t=0。
        ② **每 chunk 三文件**：wav (playable, ops afplay 可直接听) +
          -L.pcm (raw mono) + -R.pcm (raw mono)，前者独立可播放，后两者
          为最终合并提供按字节顺序拼接源。manifest (`chunkManifest`) 在
          内存维护，按 seq 排序。
        ③ **真帧驱逐**：chunker 上传成功后 `trimFramesFront` 重新分配
          inSegs/outSegs 切片，释放底层数组让 GC 回收已放置的 PCM 字节。
          对 1 小时通话 (16k 立体声) 内存峰值从 ~230MB→~O(ChunkInterval)。
        ④ **流式 Flush**：通过 `io.Pipe` + writer goroutine：
            - dry-pass 累计 manifest+tail 算总长度，写 WAV header；
            - `partReaderChain` 懒打开每个 part PCM (任意时刻只有 1 个
              HTTP body 流)，按 lKey/rKey 顺序拼成 leftReader/rightReader；
            - `streamInterleave` 按 1024-sample block 交错合并 L/R 写入
              pipe，`io.MultiWriter` 同步喂给 sha256 hasher 和 byte counter；
            - `store.Write(key, pipeReader)` 并发消费——S3/COS 等支持
              multipart 的后端会自动分片上传，本地盘流式落文件。
            - 失败回退：`pr.CloseWithError` 优雅取消 writer goroutine 防泄漏。
        ⑤ **并发安全**：chunker 与 Flush 竞态用 `r.flushed` 标志保护——
          Flush 期间 chunker 上传完检测到 flushed=true 自删 3 个 orphan
          文件，不会双放置/丢帧；Flush 自己用最新 placer state 处理 tail
          frames，效果完全等价于 chunker 接力。
        ⑥ **新增回归测试 3 个**：
          - `TestRecorder_ChunkedFlushEqualsMonolithic`：分片+流式 Flush
            vs 单次内存 Flush 在相同输入下输出大小 ±2% 一致；
          - `TestRecorder_FrameEvictionBoundedMemory`：50 帧/500ms 输入，
            chunker 跑完 inSegs/outSegs 残留 ≤10 帧（验证驱逐生效）；
          - `TestRecorder_StreamMergeFromExistingParts`：白盒构造 manifest
            + 手塞 part PCM，验证 partReaderChain 在 chunk 边界、leg-长度
            不等、L 提前耗尽等场景下逐 sample 拼接正确。
        memStore 同步加 sync.Mutex 防 chunker/Flush 并发竞态，Read 实装
        从 map 取字节流（之前是 no-op，现在 Flush 真要调）。
        全量 `go test ./pkg/voice/recorder/...` PASS；除一例与本次改动
        无关的 pkg/response/TestFriendlyErrorMessages/未知错误外全 PASS。

- [x] **voicedialog/gateway_media.go：cloudSR 默认回退到桥接率**

- [x] **voicedialog/tts_segmenter.go：playSegment 整体替换为 VS tts.Pipeline**
        用户反馈"welcome.wav 干净、上行录音干净、唯独 TTS 有电流音"——锁定到
        playSegment 自己重新发明了一遍「buffer + 分帧 + 节拍 + 兜底」，但
        welcome 用的 playPCMFrames 是从一个**连续 PCM 数组**切片，TTS player
        从 chunked channel 累积进 buffer，多重风险：
        ① emitFrame 用 time.After 而非 ticker 长话累计漂移；
        ② QCloud 偶发奇字节 chunk 直接 append → 后续帧高低字节互换 = 持续电流音；
        ③ 之前的 DC-hold 兜底每次 underrun→真 PCM 重连一次 click。
        VS 没有 tts_segmenter 这个文件，它直接用 voice/tts.Pipeline。
        改造：把 playSegment 里的 player buffer 全删，注入一个
        chanReplayService 把 prefetch 的 pcmCh 对接给 siptts.Pipeline；
        Pipeline 负责帧切分（连续 buffer 切片，chunk 边界天然安全）、
        PaceRealtime 节拍、Finalize 段尾 zero-pad。**保留** prefetch
        goroutine（段间 SDK 握手并行优化）。
        chanReplayService 同时做：
        - cloudSR != bridgeSR 时缓冲全段后**整段重采样**（避开
          InterpolatingConverter 每 chunk 重置 sourcePos 的相位错位）
        - 防御性丢弃奇字节 chunk
- [x] **voicedialog/tts_segmenter.go：移除 underrun DC-hold 兜底帧**
        bug：playSegment 在 pcmCh 暂时空时插入「最后样本复制 20ms」的兜底帧
        维持 RTP 节拍。但 hold-frame → 真 PCM 重连时 hold 住的样本值和下
        一块 PCM 首样本之间是任意阶跃（step）→ PCMA 编码后接收端 click。
        QCloud WS 推 PCM 节奏不均（80~200ms 一块），一段回复里 8-20 次
        underrun → 8-20 次 click → 用户听到的"滋滋滋"。注释里"DC hold 不
        产生波形阶跃"的自圆其说是错的，只解决了 hold 起点的阶跃，没解决
        终点。VS 的 pkg/voice/tts/Pipeline 在 underrun 时只阻塞等下一块
        PCM，从不合成兜底帧——这次彻底对齐：
        ttsUnderrunSilenceMaxMs: 200 → 0，maxSilenceFrames=0 ⇒
        select default 分支走不到 hold 路径，回落到阻塞 select。
        代价：极少数情况下 RTP cadence 短暂停顿 50~150ms，对端 SIP UA
        jitter buffer (60-200ms) 完全能吸收。
- [x] **VoiceEnvFromProcess 修死写 16k：G.711 通话不再触发任何重采样**
