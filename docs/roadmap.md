# LingEchoX 后续功能与发展方向

本文档聚焦两个问题：

1. 接下来可以做哪些“可完善”功能  
2. 项目中长期的发展方向是什么

## 1. 短期（1-2 个月）可落地功能

### 1.1 产品与业务能力补强

- **外呼活动模板化**
  - 将活动配置沉淀为模板（并发、重试、时间窗、脚本版本）。
  - 降低重复配置成本，提升运营效率。
- **联系人分群与标签**
  - 支持标签、来源、优先级、最近联络状态筛选。
  - 为精细化外呼策略做准备。
- **脚本版本管理**
  - 增加脚本版本号、发布/回滚能力。
  - 防止线上脚本误改导致业务中断。

### 1.2 工程与质量能力补强

- **统一 API 文档自动生成**
  - 建议接入 OpenAPI/Swagger 生成机制。
- **补齐关键集成测试**
  - 优先覆盖活动生命周期、呼叫状态机、脚本执行链路。
- **CI 基线**
  - 引入 `go test` + 前端 lint/type-check + 构建校验。

### 1.3 可观测性补强

- **业务指标面板**
  - 接通率、转化率、平均通话时长、失败原因分布。
- **链路追踪与告警**
  - 外呼任务异常、媒体层异常、Provider 失败率告警。

## 2. 中期（1-2 个季度）功能方向

### 2.1 智能化运营

- **智能重呼策略引擎**
  - 基于历史接通时间、意向评分、标签动态决定重呼窗口。
- **自动质检（通话语义）**
  - 通过 ASR + LLM 进行话术合规、情绪、关键意图识别。
- **知识库增强对话**
  - 结合知识库与脚本，形成问答增强型语音代理。

### 2.2 多租户与权限体系

- **组织/租户隔离**
  - 数据权限、资源额度、配置隔离。
- **细粒度 RBAC**
  - 运营、坐席、审核、管理员角色权限分离。

### 2.3 生态与扩展

- **Provider 插件化**
  - 新增 ASR/TTS/LLM 提供商时可按插件注册，降低改造成本。
- **Webhook/EventBus**
  - 将外呼事件（开始、接通、失败、结束）推送给外部系统。

## 3. 长期（年度）发展方向

### 3.1 目标产品形态

- 从“语音外呼系统”升级为“全渠道智能联络平台”：
  - 语音 + IM + 邮件 + 工单统一协同
  - 统一客户画像与交互历史
  - 全链路运营分析与自动化策略

### 3.2 数据与智能中台

- 构建语音数据资产层（脱敏、标注、检索、评估）。
- 建立模型评估体系（ASR/TTS/LLM 的准确率、时延、成本）。
- 形成业务闭环优化（运营策略自动学习与推荐）。

## 4. 建议优先级（执行顺序）

建议按以下顺序推进，避免“大而全”导致交付延迟：

1. 文档体系 + CI 基线（先保质量与可维护性）
2. 活动模板 + 脚本版本（先提运营效率）
3. 指标看板 + 告警（先增强可运营性）
4. 智能重呼 + 自动质检（再做智能化升级）
5. 多租户与插件化（最后做平台化扩展）

## 5. 里程碑示例

- **M1（4 周）**：README、架构文档、部署文档、CI 基线、核心 API 文档
- **M2（8 周）**：活动模板、脚本版本管理、核心运营报表
- **M3（12 周）**：智能重呼策略 + 初版自动质检
- **M4（16 周）**：多租户基础能力 + 事件回调能力
